# Notes: WI-022 — Startup Configuration Validation

## Context

Remote-worker accepts startup without `IDEATE_WORKER_API_KEY` and only rejects requests at runtime. Session-spawner silently treats a missing `api_key` in a worker entry as an empty string. Both behaviors mean operators discover misconfiguration from request failures rather than startup logs.

## remote-worker change

In `lifespan` (`mcp/remote-worker/server.py:85`), after the existing startup code (line 89 where `_max_concurrency` is resolved), add:

```python
if not _get_api_key():
    logger.warning(
        "IDEATE_WORKER_API_KEY is not set — all requests will be rejected with HTTP 401"
    )
```

Note: `main()` already logs `"api_key=NOT SET"` but this is at INFO level and only reachable when started via `main()`. The lifespan warning fires regardless of startup path (including tests).

## session-spawner change

In `main()` (`mcp/session-spawner/server.py`), after the `_remote_workers = valid_workers` assignment (~line 994):

```python
for w in _remote_workers:
    if not w.get("api_key"):
        logger.warning(
            "Worker '%s' has no api_key configured — remote calls to this worker will fail authentication",
            w["name"],
        )
```

## Test patterns

**remote-worker**: Using `caplog`:
```python
import logging
async def test_startup_warns_when_no_api_key(caplog, tmp_working_dir):
    with caplog.at_level(logging.WARNING, logger="outpost-remote-worker"):
        # Create client without setting IDEATE_WORKER_API_KEY
        # The lifespan warning fires when httpx ASGITransport initializes the app
        async with AsyncClient(transport=ASGITransport(app=app), ...) as client:
            pass
    assert "IDEATE_WORKER_API_KEY" in caplog.text
```

**session-spawner**: Call the `main()` startup logic or extract validation into a testable helper. The simplest approach: call `asyncio.run(main())` with a mocked stdio_server, or extract the worker validation into `_validate_workers(workers)` and test that directly. The warning uses `logger` which is `logging.getLogger(__name__)`.

## Scope

Only warnings — no behavior change. The servers continue to start normally even when misconfigured.
