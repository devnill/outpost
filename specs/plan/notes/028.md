# WI-028: Fix proc.terminate() race in cancel_job

## Context

`cancel_job` in `mcp/remote-worker/server.py` releases `job_store_lock` before calling `proc.terminate()`. In the window between lock release and signal delivery, the subprocess can exit naturally. On POSIX, calling `Popen.terminate()` on an already-dead process raises `ProcessLookupError` (ESRCH). This propagates as an unhandled exception, returning HTTP 500 instead of 204.

Tracked as OQ-011 since cycle 6. The fix is a two-line try/except.

## Location

`mcp/remote-worker/server.py` — `cancel_job` endpoint function.

The relevant block (after the lock is released) is approximately:
```python
if proc is not None:
    proc.terminate()
    try:
        await asyncio.wait_for(asyncio.to_thread(proc.wait), timeout=2.0)
    except asyncio.TimeoutError:
        proc.kill()
```

## Fix

Wrap both `proc.terminate()` and `proc.kill()` in `try/except (ProcessLookupError, OSError)`:

```python
if proc is not None:
    try:
        proc.terminate()
    except (ProcessLookupError, OSError):
        pass
    try:
        await asyncio.wait_for(asyncio.to_thread(proc.wait), timeout=2.0)
    except asyncio.TimeoutError:
        try:
            proc.kill()
        except (ProcessLookupError, OSError):
            pass
```

`OSError` is the broader catch because on some POSIX systems the error code may differ. `ProcessLookupError` is a subclass of `OSError` but being explicit about both catches the intent clearly.

## Test

Add a test in `mcp/remote-worker/test_server.py` that:
1. Creates a job with a MockPopen where `terminate()` raises `ProcessLookupError`
2. Issues `DELETE /jobs/{id}` against a running-state job
3. Asserts the response is 204 (not 500)
4. Asserts the job status is set to `cancelled`

Use the existing `MockPopen` infrastructure in the test file. Set `mock_popen.terminate.side_effect = ProcessLookupError` to simulate the race.
