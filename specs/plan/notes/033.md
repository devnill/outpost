# WI-033: Fix cancel-while-starting race in _run_claude_job

## Context

`cancel_job` reads `record.process` under `job_store_lock`. `record.process` is set in `_run_claude_job` at line 385, after `Popen` returns (line 372), running in a thread via `asyncio.to_thread`. The record status is set to `"running"` at line 452 *before* the thread starts. There is a window between the thread starting and line 385 executing where:

1. `cancel_job` acquires the lock
2. Finds `record.status == "running"` and `record.process is None`
3. Sets `record.status = "cancelled"` and returns 204
4. The thread then creates the process at line 372 and runs it to completion

The caller receives HTTP 204 (cancellation success) but the process runs uninterrupted.

WI-028 addressed the OSError race on signal delivery (process exits naturally before `proc.terminate()` is called). This work item addresses the earlier and distinct race: the process handle does not yet exist when the cancel arrives.

## Fix

In `_run_claude_job`, after `proc = subprocess.Popen(...)` returns successfully and before `record.process = proc`, add a check under `job_store_lock`:

```python
proc = subprocess.Popen(cmd, ...)

# Check if the job was cancelled while Popen was starting
with job_store_lock_sync:  # use threading.Lock, not asyncio.Lock, since this is in a sync thread
    if record.status == "cancelled":
        try:
            proc.kill()
        except (ProcessLookupError, OSError):
            pass
        return None

record.process = proc  # set before communicate() so cancel can signal it
```

Note: `job_store_lock` in `mcp/remote-worker/server.py` is an `asyncio.Lock` (`asyncio.Lock()`), which cannot be used in a sync thread context (`asyncio.to_thread` runs in a threadpool). The fix therefore needs to use a separate `threading.Lock` for the status check, or restructure using a simpler approach:

**Simpler approach (no additional lock needed)**: Read `record.status` without acquiring any lock (pure read of a string attribute is safe in CPython due to the GIL), then kill the process if cancelled:

```python
proc = subprocess.Popen(cmd, ...)

# Guard: if the job was cancelled while Popen was initializing, kill and abort
if record.status == "cancelled":
    try:
        proc.kill()
    except (ProcessLookupError, OSError):
        pass
    return None

record.process = proc
```

This is safe because: (a) `record.status` is only ever written to strings, and CPython string assignment is atomic; (b) the worst case of a race here is that we miss a cancellation and run the process anyway — which is the same behavior as today, so this check can only improve correctness, never worsen it.

## Test

In `mcp/remote-worker/test_server.py`, add a test that:

1. Creates a job and sets its status to `"running"`
2. Patches `subprocess.Popen` to record the call but return a mock process; also patches the mock process so `kill()` is trackable
3. Sets `record.status = "cancelled"` *after* the job is queued but *before* the thread assigns `record.process` — simulate this by patching `record` or by setting the status during the Popen mock call
4. Calls `_run_claude_job(record)` or `_process_job(record)` directly
5. Asserts that the mock process's `kill()` was called
6. Asserts the returned value is `None`
7. Asserts `record.status` is `"cancelled"` (not changed to `"completed"` or `"failed"`)

The simplest test: set `record.status = "cancelled"` directly before calling `_run_claude_job(record)` synchronously, verify that Popen was called, verify `proc.kill()` was called, verify return is `None`.
