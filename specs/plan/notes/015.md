# Notes: 015 — Apply Role Constraints to Remote Sessions

## Context

Source: cycle 2 review, critical finding (GP-8 violation), remote-dispatch Q-4. Decision: Option A (session-spawner resolves role at dispatch time).

Currently in `_handle_spawn_remote_session` (mcp/session-spawner/server.py around line 738–749), the payload builder passes `role` as-is (a string name) to the remote worker:

```python
if "role" in arguments and arguments["role"] is not None:
    payload["role"] = arguments["role"]
if "allowed_tools" in arguments and arguments["allowed_tools"] is not None:
    payload["allowed_tools"] = arguments["allowed_tools"]
```

The remote worker's `JobRecord` stores `role` as a label but does not look up role definitions. It has no access to `_roles` (which lives in the session-spawner process).

## Fix

Before building the payload, resolve the role if it's a string name:

```python
role_arg = arguments.get("role")
resolved_role: dict | None = None

if isinstance(role_arg, str):
    if role_arg not in _roles:
        return [TextContent(type="text", text=json.dumps({
            "error": f"Unknown role: '{role_arg}'. Available roles: {list(_roles.keys())}"
        }))]
    resolved_role = _roles[role_arg]
elif isinstance(role_arg, dict):
    resolved_role = role_arg
```

Then when building the payload, apply role defaults (caller-supplied values take priority):

```python
payload: dict = {
    "prompt": arguments["prompt"],
    "working_dir": arguments["working_dir"],
    "max_turns": arguments.get("max_turns", resolved_role.get("max_turns", DEFAULT_MAX_TURNS) if resolved_role else DEFAULT_MAX_TURNS),
    "timeout": arguments.get("timeout", DEFAULT_TIMEOUT),
    "permission_mode": arguments.get("permission_mode", resolved_role.get("permission_mode", DEFAULT_PERMISSION_MODE) if resolved_role else DEFAULT_PERMISSION_MODE),
}
if resolved_role and resolved_role.get("allowed_tools") and "allowed_tools" not in arguments:
    payload["allowed_tools"] = resolved_role["allowed_tools"]
if "allowed_tools" in arguments and arguments["allowed_tools"] is not None:
    payload["allowed_tools"] = arguments["allowed_tools"]
if role_arg:
    payload["role"] = role_arg if isinstance(role_arg, str) else role_arg.get("name", "custom")
```

Keep `role` in the payload as the label (for logging/observability in the remote worker). The key change is that `allowed_tools` and `permission_mode` from the role definition are now also included.

## Tests to Add

In `test_server.py` (session-spawner), add to the existing `TestSpawnRemoteSession` class or a new class:

1. `test_spawn_remote_session_role_name_resolves` — provide a known role name, verify the HTTP payload includes the role's `allowed_tools` and `permission_mode`
2. `test_spawn_remote_session_unknown_role_returns_error` — provide an unknown role name, verify error is returned without making an HTTP call
3. `test_spawn_remote_session_inline_role_passes_through` — provide a role as a dict, verify its fields are used in the payload

Use the existing mock pattern for `_get_http_session()` to inspect the payload posted to the worker.

## Note on _roles availability

`_roles` is a module-level global set at startup by `main()`. It is accessible in `_handle_spawn_remote_session` the same way it is in `_handle_spawn_session`. No additional plumbing needed.
