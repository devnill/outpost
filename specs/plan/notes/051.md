# WI-051 Implementation Notes: Fix server.py security, correctness, and async issues

## File to modify

`mcp/remote-worker/server.py`

## Change 1 — Fix ANTHROPIC_API_KEY secret in command list (S1)

**Location**: `_build_container_cmd` function, line 415.

**Current code**:
```python
"-e", f"ANTHROPIC_API_KEY={os.environ.get('ANTHROPIC_API_KEY', '')}",
```

**Replace with**:
```python
"-e", "ANTHROPIC_API_KEY",
```

When `docker run -e VAR` is passed a name without a value, Docker inherits the variable from the host process environment. The value is never written to the command line, so it cannot be read from `/proc/<pid>/cmdline` or `ps aux`. The remote-worker process already has `ANTHROPIC_API_KEY` in its environment (it must, or the pre-flight check below will reject the job), so Docker will find and inject it correctly.

## Change 2 — Add ANTHROPIC_API_KEY pre-flight check (M2)

**Location**: `create_job` function, after the `IDEATE_WORKER_BASE_DIR` validation block and before `job_id = str(uuid.uuid4())`.

**Add**:
```python
# Validate ANTHROPIC_API_KEY is present when container mode is active
if _agent_image and not os.environ.get("ANTHROPIC_API_KEY"):
    raise HTTPException(
        status_code=500,
        detail=(
            "ANTHROPIC_API_KEY is not set in the worker environment. "
            "Container mode requires ANTHROPIC_API_KEY to authenticate "
            "the claude CLI inside the container."
        ),
    )
```

This check only fires when `_agent_image` is non-empty. When `_agent_image` is empty (default), the check is skipped entirely and existing behavior is unchanged. Using 500 rather than 400 because this is a server misconfiguration, not a bad request from the client.

## Change 3 — Fix blocking subprocess.run in async cancel_job (AO1)

**Location**: `cancel_job` function, lines 308-316. The current blocking call:
```python
if container_name:
    try:
        subprocess.run(
            ["docker", "stop", container_name],
            timeout=15,
            capture_output=True,
        )
    except (subprocess.TimeoutExpired, FileNotFoundError, OSError):
        pass
```

**Replace with**:
```python
if container_name:
    try:
        await asyncio.to_thread(
            subprocess.run,
            ["docker", "stop", container_name],
            timeout=15,
            capture_output=True,
        )
    except (subprocess.TimeoutExpired, FileNotFoundError, OSError):
        pass
```

`cancel_job` is already `async def`, so `await asyncio.to_thread(...)` is valid here. The 15-second timeout argument is passed to `subprocess.run` (not to `to_thread`) and is respected inside the thread. The exception handling is unchanged.

## Change 4 — Add missing eviction call in cancel-while-starting path (M1)

**Location**: `_process_job` function, the `if result is None:` branch (~line 502). Current code:
```python
if result is None:
    # Cancel path: record was already marked cancelled before Popen; set duration_ms only
    async with job_store_lock:
        record.duration_ms = duration_ms
    logger.info("Job %s %s in %dms", record.job_id, record.status, duration_ms)
    return
```

**Replace with**:
```python
if result is None:
    # Cancel path: record was already marked cancelled before Popen; set duration_ms only
    async with job_store_lock:
        record.duration_ms = duration_ms
        _evict_terminal_jobs_locked()
    logger.info("Job %s %s in %dms", record.job_id, record.status, duration_ms)
    return
```

Adding `_evict_terminal_jobs_locked()` here is consistent with all three other terminal-state transitions in `_process_job` (lines 522, 542). It must be called inside the `job_store_lock` block. No other changes to this branch.
