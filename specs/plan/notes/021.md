# Notes: WI-021 — Integration Tests Between session-spawner and remote-worker

## Context

All existing tests mock either aiohttp (session-spawner) or subprocess (remote-worker). No test exercises the actual HTTP contract between the two components. This work item creates `mcp/test_integration.py` with 5 tests that use an in-process ASGI instance of the remote-worker alongside the session-spawner handlers.

## Technical Approach

The key challenge: `mcp/remote-worker` and `mcp/session-spawner` are in hyphenated directories not importable as Python packages. Use `importlib.util.spec_from_file_location` to import them.

### Import pattern

```python
import importlib.util, sys
from pathlib import Path

def _import_module(name, path):
    spec = importlib.util.spec_from_file_location(name, path)
    mod = importlib.util.module_from_spec(spec)
    sys.modules[name] = mod
    spec.loader.exec_module(mod)
    return mod

ROOT = Path(__file__).parent
worker_mod = _import_module("remote_worker", ROOT / "remote-worker" / "server.py")
spawner_mod = _import_module("session_spawner", ROOT / "session-spawner" / "server.py")
```

### In-process ASGI test setup

Use `httpx.AsyncClient` with `ASGITransport` for the remote-worker. However, the remote-worker's lifespan starts worker coroutines — we need to handle this carefully. Options:

1. **Use `httpx.AsyncClient(app=worker_mod.app)` with `base_url="http://test"`** — FastAPI lifespan runs automatically with httpx's ASGI transport (as of httpx 0.23+). Worker coroutines will start and try to drain the queue. Mock `asyncio.to_thread` or `_run_claude_job` to prevent actual subprocess calls.

2. **Simpler**: Skip the lifespan entirely by patching the lifespan on the app. Test only the HTTP request/response layer without running workers (jobs stay "queued" since worker coroutines don't start without lifespan).

For tests 1-5, option 2 works:
- Test 1: POST /jobs → returns job_id (job sits in queued state)
- Test 2: GET /jobs/{id} → returns `{"status": "queued", ...}`
- Test 3: DELETE /jobs/{id} → returns 204 (cancels queued job)
- Test 4: POST /jobs with injected prompt → GET /jobs/{id}, inspect stored prompt from job record
- Test 5: GET /jobs/nonexistent → 404

### Wiring session-spawner to the in-process worker

Rather than mocking aiohttp, wire session-spawner to use a real aiohttp session that routes through the httpx ASGI backend. httpx can act as a transport for aiohttp via a custom connector — but this is complex.

**Simpler approach**: Call the session-spawner handler functions directly with `_remote_workers` configured to point at a test httpx server. Start the FastAPI app with uvicorn on a random port in an asyncio background task, configure `spawner_mod._remote_workers` to that port, use a real aiohttp ClientSession.

```python
@pytest.fixture
async def worker_server():
    import uvicorn
    config = uvicorn.Config(worker_mod.app, host="127.0.0.1", port=0, lifespan="off")
    server = uvicorn.Server(config)
    task = asyncio.create_task(server.serve())
    # wait for server to be ready...
    yield server
    server.should_exit = True
    await task
```

**Simplest workable approach**: Use `httpx` as an ASGI test client for the worker, and for the session-spawner side, mock `_get_http_session()` to return an aiohttp-compatible async context manager that delegates to the httpx client. This requires an adapter.

**Recommended**: Use the `pytest-httpserver` or `aioresponses` approach, but those add dependencies. Instead, set up a real uvicorn server on a free port (using `port=0` and reading the assigned port back), configure `_remote_workers` to point at it, and use a real `aiohttp.ClientSession`. This tests the full HTTP path end-to-end.

## Test 4 Implementation

For the system_prompt test: after submitting a job with `role="reviewer"`, read `job_store[job_id].prompt` from the worker module's global state. Since worker_mod is the imported module, `worker_mod.job_store[job_id].prompt` gives the stored prompt. Assert it starts with `[ROLE: reviewer]`.
