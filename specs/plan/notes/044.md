## CF2: Test _worker drops cancelled-while-queued job

Path in `_worker` (server.py lines ~483–506):
```python
async with job_store_lock:
    record = job_store.get(job_id)
    if not record or record.status != "queued":
        # Job was cancelled or removed while queued
        continue
```

Test strategy: insert a job record with `status = "cancelled"` into `job_store`, enqueue its `job_id` via `put_nowait`, then run one iteration of the worker. Verify `_process_job` is never called.

Use `unittest.mock.patch` to wrap `_process_job` and confirm it is not called:
```python
async def test_worker_skips_cancelled_while_queued_job(tmp_working_dir):
    job_id = str(uuid.uuid4())
    record = worker_mod.JobRecord(job_id, ...)
    record.status = "cancelled"
    async with worker_mod.job_store_lock:
        worker_mod.job_store[job_id] = record
    worker_mod.job_queue.put_nowait(job_id)

    with patch.object(worker_mod, "_process_job") as mock_process:
        # Run worker for one iteration then cancel it
        task = asyncio.create_task(worker_mod._worker(0))
        await asyncio.sleep(0.05)  # allow one queue.get() + continue
        task.cancel()
        try:
            await task
        except asyncio.CancelledError:
            pass

    mock_process.assert_not_called()
    async with worker_mod.job_store_lock:
        assert worker_mod.job_store[job_id].status == "cancelled"
```

Note: the `_reset_globals` fixture clears job_store between tests.

## CF3: Test _process_job cancel-while-starting path (result is None)

The cancel-while-starting path in `_process_job` (server.py line ~430–435):
```python
if result is None:
    # Cancel path: record was already marked cancelled before Popen; set duration_ms only
    async with job_store_lock:
        record.duration_ms = duration_ms
    return
```

Test strategy: create a job record with `status = "cancelled"` (already cancelled), patch `_run_claude_job` to return `None`, call `_process_job` directly, verify only `duration_ms` is set.

```python
async def test_process_job_cancel_while_starting_sets_only_duration(tmp_working_dir):
    job_id = str(uuid.uuid4())
    record = worker_mod.JobRecord(job_id, ...)
    record.status = "cancelled"
    async with worker_mod.job_store_lock:
        worker_mod.job_store[job_id] = record

    with patch.object(worker_mod, "_run_claude_job", return_value=None):
        await worker_mod._process_job(record)

    assert record.duration_ms is not None
    assert record.status == "cancelled"       # not overwritten
    assert record.completed_at is None        # not set by _process_job (was already set by cancel_job)
    assert record.exit_code is None
    assert record.output is None
```

Note: in the real cancellation flow, `cancel_job` sets `completed_at` on the record before signalling the process. `_process_job` must not overwrite it. This test verifies `_process_job` leaves `completed_at` alone on the cancel path.

## CF5: Test cancel_job SIGKILL fallback

The SIGKILL fallback in `cancel_job` (server.py lines ~300–309):
```python
try:
    await asyncio.wait_for(asyncio.to_thread(proc.wait), timeout=2.0)
except asyncio.TimeoutError:
    try:
        proc.kill()
    except (ProcessLookupError, OSError):
        pass
```

Test strategy: patch `asyncio.wait_for` to raise `asyncio.TimeoutError` so the SIGKILL branch is exercised.

```python
async def test_cancel_job_sigkill_fallback_on_timeout(client, api_key_env, tmp_working_dir):
    # Submit job
    resp = await client.post("/jobs", json={"prompt": "x", "working_dir": str(tmp_working_dir)},
                              headers={"X-API-Key": "key"})
    job_id = resp.json()["job_id"]

    # Manually set the job to running with a mock process
    mock_proc = MagicMock()
    async with worker_mod.job_store_lock:
        record = worker_mod.job_store[job_id]
        record.status = "running"
        record.process = mock_proc

    # Patch asyncio.wait_for to raise TimeoutError (simulates proc.wait() not returning in 2s)
    with patch("asyncio.wait_for", side_effect=asyncio.TimeoutError):
        resp = await client.delete(f"/jobs/{job_id}", headers={"X-API-Key": "key"})

    assert resp.status_code == 204
    mock_proc.terminate.assert_called_once()
    mock_proc.kill.assert_called_once()
```

Alternatively, avoid patching `asyncio.wait_for` globally (which could interfere with other async ops) by patching the module-level reference or using a shorter approach: mock `proc.wait` in `asyncio.to_thread` to block indefinitely, and set a very short timeout in the actual code. But patching `asyncio.wait_for` in the `cancel_job` endpoint's module is the most direct approach.

Check whether `asyncio.wait_for` is imported directly in the module or called as `asyncio.wait_for`. If called as `asyncio.wait_for`, patch `mcp.remote-worker.server.asyncio.wait_for` or patch the `cancel_job` function's reference. In the code it's `await asyncio.wait_for(...)` so patching `worker_mod.asyncio.wait_for` (via `patch.object(worker_mod.asyncio, "wait_for", ...)`) is the right approach.
