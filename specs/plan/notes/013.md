# Notes: 013 — Fix poll_remote_job Auth-Error Priority

## Context

Source: cycle 2 review, significant finding S1, remote-dispatch Q-3.

In `_handle_poll_remote_job` (mcp/session-spawner/server.py around line 852–882), there are two sequential passes over the `poll_results` list:

1. First pass: return immediately if any worker returned an auth error (status `auth_error`)
2. Second pass: return if any worker returned a found result (status `found`)

This ordering means that a single misconfigured worker with a stale API key blocks all poll results, even if another worker actually has the job. This violates GP-3 (Graceful Degradation).

## Fix

Swap the order of the two passes:

1. First pass: return the first found result
2. Second pass: if no found result, return auth error if any exists
3. Fallback: not-found error

The fix is literally swapping the two for-loop blocks. The logic within each block is unchanged.

## Code Location

Look for the comment `# First pass: return auth errors immediately (definitive)` — this is where the swap happens.

After the swap, the comment for the new first pass should read: `# First pass: return found result`
The second pass comment: `# Second pass: return auth error if no result found`

## Verification

The existing test `test_poll_remote_job_auth_error_worker` may need to be updated if it tests the current (wrong) behavior where auth error blocks a found result. Check the test and update if needed.
