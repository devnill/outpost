# WI-054 Implementation Notes: Add docker stop cancel path test

## File to modify

`mcp/remote-worker/test_server.py`

## What to add

Add a new async test that verifies `cancel_job` calls `docker stop job-{job_id}` when cancelling a running containerized job. Place the test near the other cancel tests (around the existing `test_cancel_running_job_*` tests).

## Test setup

The test needs:
1. A job record in "running" state with `container_name` set
2. A mock process object (so `proc.terminate()` and `proc.wait` don't fail)
3. A monkeypatched `subprocess.run` to capture docker stop calls (via `asyncio.to_thread`)
4. An HTTP client to call `DELETE /jobs/{job_id}`

## Implementation pattern

```python
@pytest.mark.asyncio
async def test_cancel_running_container_job_calls_docker_stop(monkeypatch):
    """cancel_job issues docker stop when container_name is set on a running job."""
    import uuid
    from unittest.mock import MagicMock

    job_id = str(uuid.uuid4())
    container_name = f"job-{job_id}"

    # Create a running job with container_name set
    req = worker.JobRequest(prompt="test", working_dir="/tmp")
    record = worker.JobRecord(job_id, req)
    record.status = "running"
    record.container_name = container_name

    # Mock process: terminate() and wait() must not raise
    mock_proc = MagicMock()
    mock_proc.wait.return_value = None  # called via asyncio.to_thread(proc.wait)
    record.process = mock_proc

    async with worker.job_store_lock:
        worker.job_store[job_id] = record

    # Capture subprocess.run calls (called via asyncio.to_thread in cancel_job)
    docker_stop_calls = []
    original_run = worker.subprocess.run

    def fake_subprocess_run(cmd, **kwargs):
        docker_stop_calls.append(list(cmd))
        return MagicMock(returncode=0)

    monkeypatch.setattr(worker.subprocess, "run", fake_subprocess_run)

    try:
        async with AsyncClient(app=worker.app, base_url="http://test") as ac:
            monkeypatch.setattr(worker, "_get_api_key", lambda: "test-key")
            response = await ac.delete(
                f"/jobs/{job_id}",
                headers={"X-API-Key": "test-key"},
            )
        assert response.status_code == 204

        # Verify docker stop was called with the correct container name
        assert any(
            cmd[:3] == ["docker", "stop", container_name]
            for cmd in docker_stop_calls
        ), f"Expected docker stop {container_name!r} call, got: {docker_stop_calls}"
    finally:
        async with worker.job_store_lock:
            worker.job_store.pop(job_id, None)
```

## Important notes

- `cancel_job` now uses `await asyncio.to_thread(subprocess.run, ...)` (after WI-051). Monkeypatching `worker.subprocess.run` intercepts this call because `asyncio.to_thread` calls the function object, and the monkeypatched version is the function object that will be called in the thread.
- `mock_proc.wait` must be a callable that returns immediately (not a coroutine). `asyncio.to_thread(proc.wait)` will call `proc.wait()` in a thread pool. `MagicMock().wait` returns a MagicMock which is callable and returns immediately — this is sufficient.
- The `_get_api_key` monkeypatch is needed because the test uses a real FastAPI app (via `httpx.AsyncClient`) which goes through the auth middleware.
- Check the existing cancel tests in `test_server.py` for the `AsyncClient` import and `_get_api_key` patching pattern — they likely already use `AsyncClient` from `httpx` or the `TestClient` from `starlette`.

## Note on _reset_globals

If using the `_reset_globals` autouse fixture, make sure the job_store cleanup in the test's `finally` block is redundant (the fixture handles it). But always clean up within the test to be safe.
