## NG4: Fix submit_job store ordering (create_job endpoint, server.py lines ~174–214)

Current order:
```python
job_id = str(uuid.uuid4())
record = JobRecord(job_id, request)
try:
    job_queue.put_nowait(job_id)
except asyncio.QueueFull:
    raise HTTPException(status_code=429, detail="Job queue is full. Try again later.")
async with job_store_lock:
    job_store[job_id] = record
```

Correct order (store-then-enqueue):
```python
job_id = str(uuid.uuid4())
record = JobRecord(job_id, request)
async with job_store_lock:
    job_store[job_id] = record
try:
    job_queue.put_nowait(job_id)
except asyncio.QueueFull:
    async with job_store_lock:
        del job_store[job_id]
    raise HTTPException(status_code=429, detail="Job queue is full. Try again later.")
```

Rationale: a worker coroutine that dequeues job_id immediately after put_nowait must find the record in job_store. The current order has a window (within the same coroutine frame, so no actual race under asyncio) where the record is not yet stored. The fix is semantically correct; the rollback path handles the QueueFull case cleanly.

Current list_jobs entry:
```python
entry = {
    "job_id": record.job_id,
    "status": record.status,
    "role": record.role,
    "created_at": record.created_at,
}
if record.duration_ms is not None:
    entry["duration_ms"] = record.duration_ms
```

Add started_at and completed_at conditionally (only include when not None):
```python
if record.started_at is not None:
    entry["started_at"] = record.started_at
if record.completed_at is not None:

```

These fields exist on JobRecord (set in _worker when status → "running", and in cancel_job / _process_job when terminal). They are already returned by GET /jobs/{id}; list_jobs should include them for consistency.
