# WI-031: Add --cwd flag to _run_claude_job; add max_jobs to list_remote_workers output

## Context

Two independent one-line fixes identified in cycle 7:

1. **--cwd flag (OQ-023)**: `_run_claude_job` in remote-worker sets `cwd=record.working_dir` on the subprocess but does not pass `--cwd` to the Claude CLI. Session-spawner passes both. The `--cwd` flag controls project root detection, CLAUDE.md loading, and trust boundary enforcement. The docstring falsely claims the two servers use "the same subprocess pattern."

2. **max_jobs field (OQ-022)**: `_fetch_worker_health` in session-spawner does not forward `max_jobs` from the remote-worker `/health` response, despite the field being specified in architecture.md section 3 and available in the worker response. One-line fix at `mcp/session-spawner/server.py:684`.

## Fix 1: --cwd in remote-worker

In `mcp/remote-worker/server.py`, in `_run_claude_job`, locate the `cmd` list construction. It currently includes `"--max-turns"` and other flags. Add `"--cwd"` and `record.working_dir` to the command list.

Before (approximate):
```python
cmd = [
    "claude",
    "--print",
    "--output-format", output_format,
    "--max-turns", str(record.max_turns),
    record.prompt,
]
```

After:
```python
cmd = [
    "claude",
    "--print",
    "--output-format", output_format,
    "--max-turns", str(record.max_turns),
    "--cwd", record.working_dir,
    record.prompt,
]
```

Also update the docstring to remove the false equivalence claim. Replace "Uses the same subprocess pattern as session-spawner" with an accurate description of what the function does.

## Fix 2: max_jobs in _fetch_worker_health

In `mcp/session-spawner/server.py`, in `_fetch_worker_health`, find the dict returned on a successful health response. Add `"max_jobs": data.get("max_jobs")` to the returned dict.

Current returned dict includes: `name`, `url`, `status`, `active_jobs`, `queued_jobs`, `max_concurrency`.
Add: `"max_jobs": data.get("max_jobs")`.
