# Notes: WI-020 — LRU Eviction for Job Store

## Context

`job_store: dict[str, JobRecord]` in `mcp/remote-worker/server.py` grows without bound. Jobs are never removed after reaching terminal state (completed, failed, cancelled). This causes a slow memory leak for long-running worker daemons.

## Implementation

### Module-level variable

Add alongside `_max_concurrency`:
```python
_max_jobs: int = 1000
```

### Configuration reading

In `lifespan` (line 85), alongside `_max_concurrency = _get_max_concurrency()`:
```python
global _max_jobs
try:
    _max_jobs = int(os.environ.get("IDEATE_WORKER_MAX_JOBS", "1000"))
except ValueError:
    _max_jobs = 1000
```

### Eviction function

```python
def _evict_terminal_jobs() -> None:
    """Remove oldest terminal jobs when job_store exceeds _max_jobs. Called under job_store_lock."""
    if len(job_store) <= _max_jobs:
        return
    terminal = [r for r in job_store.values() if r.status in ("completed", "failed", "cancelled")]
    # Sort by completion timestamp (fallback to creation time for cancelled-while-queued)
    terminal.sort(key=lambda r: r.completed_at or r.created_at or "")
    evict_count = len(job_store) - _max_jobs
    for record in terminal[:evict_count]:
        del job_store[record.job_id]
```

### Call sites (while holding job_store_lock)

1. In `_process_job` (around line 363), after setting `record.status`:
   ```python
   async with job_store_lock:
       record.output = output
       record.exit_code = exit_code
       ...
       if record.status != "cancelled":
           record.completed_at = completed_at
           record.status = "failed" if exit_code != 0 else "completed"
       _evict_terminal_jobs()  # ADD THIS
   ```

2. In `cancel_job` (line 257 for queued, line 262 for running), after setting status:
   After the `async with job_store_lock` block sets `record.status = "cancelled"`, call `_evict_terminal_jobs()` inside the lock.

   For the running case, eviction must happen inside the lock before releasing it (the `proc = record.process` capture and status set are inside the lock already). Add `_evict_terminal_jobs()` before the `return None` / end of the lock block.

### Health endpoint

Add `"max_jobs": _max_jobs` to the health response dict:
```python
return {
    "status": "ok",
    "version": VERSION,
    "active_jobs": active,
    "queued_jobs": queued,
    "max_concurrency": _max_concurrency,

```

## Test Approach

Set `_max_jobs = 2` (override the module-level variable directly in a test), add 3 completed jobs, assert the oldest is evicted. Use the `_execute_job` helper already in the test file to drive jobs to completion.
