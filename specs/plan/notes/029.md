# WI-029: Handle FileNotFoundError for missing claude binary

## Context

Both servers call `claude` as a subprocess. If `claude` is not on PATH (most common new-user error), both servers currently surface a raw `FileNotFoundError`:
- session-spawner: propagates through `asyncio.to_thread` as an unhandled exception â†’ MCP protocol error with Python traceback
- remote-worker: caught by the generic exception handler in `_worker`, marks job `failed` with raw exception text

Tracked as OQ-024 / Q-6 since cycle 3. Policy P-7 (added in cycle 7 domain update) states: subprocess call sites must catch missing-binary errors and return actionable messages.

## session-spawner Fix

In `mcp/session-spawner/server.py`, the `claude` subprocess is invoked via `asyncio.to_thread(subprocess.run, ["claude", ...], ...)`.

Locate the `asyncio.to_thread` call inside the `_handle_spawn_session` handler. Wrap it to catch `FileNotFoundError`:

```python
try:
    result = await asyncio.to_thread(subprocess.run, cmd, ...)
except FileNotFoundError:
    return [TextContent(type="text", text=json.dumps({
        "error": "claude CLI not found on PATH. Ensure Claude Code is installed and the 'claude' binary is accessible in the server process environment.",
        "exit_code": 1
    }))]
```

The return format must match the existing error response structure used elsewhere in the handler (json-serialized dict with `error` and `exit_code` keys).

## remote-worker Fix

In `mcp/remote-worker/server.py`, the `claude` subprocess is invoked in `_run_claude_job` via `subprocess.Popen(cmd, ...)`.

Add a `FileNotFoundError` handler around the `Popen` call:

```python
try:
    proc = subprocess.Popen(cmd, ...)
except FileNotFoundError:
    record.status = "failed"
    record.error = "claude CLI not found on PATH. Ensure Claude Code is installed and the 'claude' binary is accessible in the server process environment."
    record.completed_at = _now()
    return
```

Set `record.status = "failed"` and `record.error` to the actionable message, then return early. Do not set `record.output` or `record.exit_code` (these remain None for a launch-failed job).

- No Python exception traceback appears

- The job reaches `failed` status
- `job.error` contains "PATH" and "claude"
- The endpoint returns the job with `status: failed` and the error message
