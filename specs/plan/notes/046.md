## NG3: Auth layer not exercised in MCP handler integration test flows

The existing integration tests exercise HTTP directly (with `headers={"X-API-Key": "test-key"}`) but the MCP handler tests (`_handle_poll_remote_job`, `_handle_cancel_remote_job`) go through the spawner's aiohttp session which also sends the `X-API-Key` header. NG3 notes that no test verifies what happens when the api_key is wrong through these MCP handler paths.

The `worker_server` fixture (test_integration.py lines ~41–98) patches `worker_mod._get_api_key = lambda: "test-key"` and sets `spawner_mod._remote_workers` with `api_key: "test-key"`. To test auth failure, use a variant fixture or inline setup that gives the server a different key than the spawner sends.

## Test 1: Wrong api_key causes auth error through MCP spawn handler

```python
@pytest.mark.asyncio
async def test_spawn_remote_session_wrong_api_key_returns_auth_error(worker_server, http_session):
    _server, port = worker_server

    # Override the spawner's worker config with a wrong api_key
    # The worker fixture sets the server to expect "test-key"
    spawner_mod._remote_workers = [
        {
            "name": "test-worker",
            "url": f"http://127.0.0.1:{port}",
            "api_key": "wrong-key",   # deliberately wrong
        }
    ]

    result = await spawner_mod._handle_spawn_remote_session({
        "prompt": "hello",
        "working_dir": _working_dir(),
    })
    data = json.loads(result[0].text)
    assert "error" in data
    assert (
        "auth" in data["error"].lower()
        or "401" in data["error"]
        or "invalid" in data["error"].lower()
        or "key" in data["error"].lower()
    )
```

Note: the spawner's health check also uses the api_key. `_fetch_worker_health` sends a GET /health which is NOT authenticated (health endpoint skips auth middleware). So the wrong key does not block worker selection — only the POST /jobs call returns 401.

Check whether `_handle_spawn_remote_session` handles a 401 from POST /jobs gracefully (returns an error TextContent vs raises an exception). If it currently propagates the exception, the test will fail — in that case, also verify the error handling path exists and add a fix if needed.

## Test 2: Correct api_key succeeds through MCP spawn handler (complements existing tests)

```python
@pytest.mark.asyncio
async def test_spawn_remote_session_correct_key_returns_job_id(worker_server, http_session):
    """Exercises the full MCP handler path including auth header forwarding."""
    result = await spawner_mod._handle_spawn_remote_session({
        "prompt": "hello",
        "working_dir": _working_dir(),
        "worker_name": "test-worker",
    })
    data = json.loads(result[0].text)
    assert "job_id" in data
    assert data["status"] == "queued"
```

This test uses `worker_name` to bypass the health check (no least-loaded selection) and goes straight to POST /jobs with the correct key. It verifies the full MCP-layer auth forwarding path.

## Fixture consideration

The `worker_server` fixture restores `spawner_mod._remote_workers = []` on teardown (line 96). If Test 1 overrides `_remote_workers`, the fixture's teardown will still reset it cleanly. No fixture modification needed.

The `http_session` fixture injects `spawner_mod._http_session` (line 105). Tests that call MCP handlers need this fixture to be active so the spawner has an http session.

Both tests above need both `worker_server` and `http_session` fixtures (in that order, since `http_session` depends on `worker_server`).
